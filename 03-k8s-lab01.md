## Seção 1 — O que é Kubernetes e por que usar?

Imagine que você tem uma aplicação rodando dentro de um contêiner Docker. Funciona bem. Mas agora imagine que essa aplicação precisa:

- Rodar em **várias cópias** para suportar mais usuários
- **Reiniciar sozinha** se uma cópia falhar
- Ser **atualizada** sem ficar fora do ar

Fazer tudo isso manualmente com `docker run` seria um pesadelo. É aqui que o Kubernetes entra.

### A analogia do gerente de hotel

Pense no Kubernetes como o **gerente de um hotel**:

- Você (o hóspede) diz: *"Preciso de 3 quartos limpos, com ar-condicionado, no 5º andar."*
- Você **não** diz como limpar os quartos, nem qual funcionário deve fazer o quê.
- O gerente (Kubernetes) **cuida de tudo**: atribui quartos, chama a limpeza, e se um ar-condicionado quebrar, providencia outro quarto.

No Kubernetes, você declara o **estado desejado** ("quero 3 cópias da minha aplicação rodando") e ele se encarrega de tornar isso realidade — e de manter assim.

## Seção 2 — Habilitando o Kubernetes no Docker Desktop

> ⚠️ **ATENÇÃO:** O Kubernetes **NÃO vem habilitado por padrão** no Docker Desktop. Você precisa ativá-lo manualmente. Sem esse passo, nenhum comando deste laboratório funcionará.

### Passo 1 — Verificar recursos da máquina

O Kubernetes consome memória e CPU. Antes de habilitá-lo, garanta que o Docker Desktop tem recursos suficientes.

**macOS / Windows (sem WSL2):**

1. Abra o Docker Desktop
2. Clique no ícone de engrenagem (**Settings/Preferências**)
3. Vá em **Resources**
4. Configure:
   - **CPUs:** mínimo 2 (recomendado: 4)
   - **Memory:** mínimo 4 GB (recomendado: 6 GB)
5. Clique em **Apply & Restart**

**Windows com WSL2:**

No modo WSL2, o Docker Desktop não mostra controles de CPU/memória na interface. O controle é feito por um arquivo de configuração do próprio WSL2.

1. Abra o PowerShell ou o Prompt de Comando
2. Crie ou edite o arquivo `C:\Users\SEU_USUARIO\.wslconfig`:

```ini
[wsl2]
memory=6GB
processors=4
```

3. Reinicie o WSL2:

```powershell
wsl --shutdown
```

4. Reabra o Docker Desktop

**Linux:**

No Linux, o Docker Desktop usa os recursos da máquina diretamente. Se sua máquina tem pelo menos 4 GB de RAM livres, você está bem.

### Passo 2 — Habilitar o Kubernetes

1. Abra o Docker Desktop
2. Vá em **Settings** (ícone de engrenagem)
3. No menu lateral, clique em **Kubernetes**
4. Marque a caixa **Enable Kubernetes**
5. Clique em **Apply & Restart**
6. **Aguarde** — a primeira ativação pode levar de 2 a 5 minutos. O ícone do Kubernetes no canto inferior ficará verde quando estiver pronto.

### Passo 3 — Verificar que funcionou

Abra seu terminal e execute:

```bash
kubectl version
# Exibe a versão do cliente (kubectl) e do servidor (cluster Kubernetes).
# Se ambos aparecerem, o Kubernetes está rodando.
```

Se você vir algo como `Unable to connect to the server`, o Kubernetes ainda está iniciando — aguarde mais um pouco e tente novamente.

## Seção 3 — Entendendo o kubectl e o kubeconfig

Antes de executar qualquer comando, você precisa entender **duas peças fundamentais**: o `kubectl` e o arquivo `kubeconfig`.

### O que é o kubectl?

O `kubectl` (pronuncia-se "kube-control" ou "kube-c-t-l") é a **ferramenta de linha de comando** do Kubernetes. Ele é a sua forma principal de conversar com o cluster.

Tudo que você faz no Kubernetes — criar aplicações, verificar status, diagnosticar problemas — passa pelo `kubectl`.

### O que é o kubeconfig?

Quando você digita um comando `kubectl`, ele precisa saber **para onde enviar** esse comando. Essa informação fica em um arquivo chamado `kubeconfig`.

O arquivo `kubeconfig` contém:

- **Clusters:** endereços de clusters Kubernetes que você pode acessar
- **Usuários:** credenciais para se autenticar em cada cluster
- **Contextos:** combinações de cluster + usuário + namespace (explicaremos a seguir)

Por padrão, o arquivo fica em:

| Sistema | Caminho |
|---|---|
| macOS / Linux | `~/.kube/config` |
| Windows | `C:\Users\SEU_USUARIO\.kube\config` |

Quando você habilitou o Kubernetes no Docker Desktop, ele **criou automaticamente** esse arquivo (ou adicionou uma entrada nele) para que o `kubectl` saiba como se conectar ao cluster local.

### Verificando na prática

```bash
kubectl config view
# Mostra o conteúdo do seu kubeconfig.
# Você deve ver um cluster chamado "docker-desktop"
# e um contexto com o mesmo nome.
```

## Seção 4 — Contextos: para onde seus comandos vão

### O que é um contexto?

Um contexto é um **atalho nomeado** dentro do kubeconfig que combina três informações:

1. **Cluster** — qual cluster Kubernetes você quer acessar
2. **Usuário** — com quais credenciais
3. **Namespace** — em qual "divisão" dentro do cluster (opcional)

Por que isso importa? Porque em um ambiente real, você pode ter vários clusters configurados (desenvolvimento, staging, produção). O contexto ativo determina **para onde todos os seus comandos vão**.

> ⚠️ **CUIDADO:** Executar comandos no contexto errado é um dos erros mais perigosos no Kubernetes. Imagine deletar recursos no cluster de produção achando que está no de desenvolvimento. Sempre verifique seu contexto antes de executar comandos destrutivos.

### Laboratório 1 — Trabalhando com contextos

**1. Ver o contexto ativo:**

```bash
kubectl config current-context
# Mostra o nome do contexto que está ativo agora.
# Deve retornar: docker-desktop
```

**2. Listar todos os contextos disponíveis:**

```bash
kubectl config get-contexts
# Lista todos os contextos do seu kubeconfig.
# O asterisco (*) na coluna CURRENT indica qual está ativo.
```

**3. Trocar de contexto (para referência futura):**

Neste laboratório temos apenas um contexto, mas o comando para trocar é:

```bash
kubectl config use-context docker-desktop
# Alterna o contexto ativo para "docker-desktop".
# Útil quando você tem múltiplos clusters configurados.
```

## Seção 5 — Namespaces: divisões dentro do cluster

Antes de criarmos qualquer recurso, é importante entender o conceito de **namespace**.

### O que é um namespace?

Um namespace é uma **divisão lógica** dentro de um cluster Kubernetes. Pense nele como pastas no seu computador — os arquivos de trabalho ficam separados dos arquivos pessoais, mas todos estão no mesmo disco.

Namespaces servem para:

- Separar ambientes (ex.: `desenvolvimento`, `staging`)
- Separar equipes (ex.: `equipe-frontend`, `equipe-backend`)
- Evitar conflitos de nomes entre recursos

### Namespaces padrão

Todo cluster Kubernetes já nasce com alguns namespaces:

- `default` — onde seus recursos vão se você não especificar outro
- `kube-system` — onde ficam os componentes internos do Kubernetes (não mexa aqui)
- `kube-public` — recursos públicos do cluster (raramente usado diretamente)

### Laboratório 2 — Explorando namespaces

**1. Listar os namespaces existentes:**

```bash
kubectl get namespaces
# Mostra todos os namespaces do cluster.
# Você verá pelo menos: default, kube-system, kube-public
```

**2. Criar um namespace para este laboratório:**

```bash
kubectl create namespace laboratorio
# Cria um novo namespace chamado "laboratorio".
# Vamos usar este namespace para manter nossos recursos organizados.
```

**3. Confirmar a criação:**

```bash
kubectl get namespaces
# Agora o namespace "laboratorio" deve aparecer na lista.
```

A partir de agora, usaremos a flag `-n laboratorio` nos comandos para direcionar nossos recursos a esse namespace.

## Seção 6 — Pods: a menor unidade do Kubernetes

### O que é um Pod?

Um Pod é a **menor unidade que o Kubernetes gerencia**. Ele é um envelope que contém um ou mais contêineres.

"Mas por que não gerenciar os contêineres diretamente?"

Porque o Kubernetes precisa de uma camada extra para gerenciar coisas que contêineres isolados não sabem fazer: compartilhar rede entre si, compartilhar volumes de armazenamento, e ser agendados juntos no mesmo nó. O Pod é essa camada.

**Na prática para iniciantes:** na grande maioria dos casos, um Pod contém **exatamente um contêiner**. Pense em Pod como sinônimo de "um contêiner rodando no Kubernetes", e você estará certo em 90% das situações.

### Pod vs. Contêiner — Qual a diferença?

| | Contêiner | Pod |
|---|---|---|
| **O que é** | Um processo isolado rodando uma aplicação | Um envelope que contém um ou mais contêineres |
| **Gerenciado por** | Docker (ou outro runtime) | Kubernetes |
| **Tem IP próprio?** | Não (compartilha com o Pod) | Sim |
| **Criado com** | `docker run` | `kubectl` ou manifesto YAML |

### De onde vêm as imagens?

Quando criamos um Pod, dizemos qual **imagem de contêiner** ele deve rodar. Essas imagens vêm de **registries** (repositórios de imagens). Os mais comuns são:

- **Docker Hub** (`docker.io`) — o registry padrão. Se você escrever `nginx`, o Kubernetes busca `docker.io/library/nginx`
- **GitHub Container Registry** (`ghcr.io`)
- **Registries privados** de empresas

Para este laboratório, usaremos imagens públicas do Docker Hub.

### Laboratório 3 — Criando seu primeiro Pod

Vamos criar um Pod rodando o Nginx (um servidor web leve e popular).

**1. Criar o Pod:**

```bash
kubectl run meu-nginx \
  --image=nginx:latest \
  -n laboratorio
# kubectl run       → cria um Pod
# meu-nginx         → nome que estamos dando ao Pod
# --image=nginx     → imagem do contêiner (vem do Docker Hub)
# -n laboratorio    → cria no namespace "laboratorio"
```

**2. Verificar que o Pod foi criado:**

```bash
kubectl get pods -n laboratorio
# Lista todos os Pods no namespace "laboratorio".
# A coluna STATUS deve mostrar "Running" após alguns segundos.
# Se mostrar "ContainerCreating", aguarde e execute novamente.
```

**3. Ver detalhes do Pod:**

```bash
kubectl describe pod meu-nginx -n laboratorio
# Mostra informações detalhadas sobre o Pod:
# - Em qual nó está rodando
# - Qual imagem está usando
# - Eventos (downloads de imagem, início do contêiner, erros)
# Útil para diagnosticar problemas.
```

**4. Acessar os logs do contêiner:**

```bash
kubectl logs meu-nginx -n laboratorio
# Mostra a saída padrão (stdout) do contêiner dentro do Pod.
# Para o Nginx, você verá as mensagens de inicialização.
```

**5. Executar um comando dentro do contêiner:**

```bash
kubectl exec -it meu-nginx -n laboratorio -- /bin/bash
# Abre um terminal interativo dentro do contêiner.
# -i  → modo interativo (mantém stdin aberto)
# -t  → aloca um pseudo-terminal (para você poder digitar)
# --  → separa os argumentos do kubectl dos argumentos do comando
# /bin/bash → o shell que queremos abrir dentro do contêiner
```

Uma vez dentro do contêiner, explore:

```bash
hostname
# Mostra o nome do host (será o nome do Pod)

curl localhost
# Faz uma requisição HTTP ao Nginx rodando neste mesmo contêiner.
# Você deve ver o HTML da página padrão do Nginx.

exit
# Sai do contêiner e volta ao seu terminal local.
```

**6. Deletar o Pod:**

```bash
kubectl delete pod meu-nginx -n laboratorio
# Remove o Pod. No Kubernetes, Pods criados diretamente com "kubectl run"
# não são recriados automaticamente — uma vez deletados, somem para sempre.
# (Deployments resolvem isso, como veremos adiante.)
```

## Seção 7 — YAML: a linguagem dos manifestos Kubernetes

Até agora usamos comandos `kubectl run` para criar recursos. Isso funciona para testes rápidos, mas no dia a dia do Kubernetes, **tudo é definido em arquivos YAML**.

### Por que YAML?

- É **declarativo**: você descreve o estado desejado, não os passos para chegar lá
- É **versionável**: arquivos YAML podem ser guardados no Git
- É **reproduzível**: qualquer pessoa pode aplicar o mesmo arquivo e obter o mesmo resultado

### Sintaxe básica do YAML

Antes de escrever manifestos Kubernetes, você precisa entender a sintaxe do YAML. Não é difícil, mas tem regras que devem ser seguidas à risca.

**Regra 1 — Chave e valor, separados por dois-pontos:**

```yaml
nome: João
idade: 30
```

**Regra 2 — Hierarquia é definida por indentação (espaços, NUNCA tabs):**

```yaml
endereco:
  rua: Av. Paulista
  numero: 1000
  cidade: São Paulo
```

Aqui, `rua`, `numero` e `cidade` são "filhos" de `endereco` porque estão indentados com 2 espaços.

> ⚠️ **YAML é extremamente sensível à indentação.** Um espaço a mais ou a menos causa erros. Use sempre **2 espaços** por nível. Nunca use a tecla Tab.

**Regra 3 — Listas usam um hífen (`-`) antes de cada item:**

```yaml
frutas:
  - maçã
  - banana
  - laranja
```

**Regra 4 — Listas de objetos combinam hífens com chave-valor:**

```yaml
pessoas:
  - nome: Ana
    idade: 25
  - nome: Carlos
    idade: 32
```

O hífen indica um novo item da lista, e as chaves abaixo dele pertencem àquele item.

### Anatomia de um manifesto Kubernetes

Todo recurso no Kubernetes segue esta estrutura:

```yaml
apiVersion: v1            # Versão da API do Kubernetes para este tipo de recurso
kind: Pod                 # Tipo de recurso (Pod, Deployment, Service, etc.)
metadata:                 # Informações SOBRE o recurso
  name: meu-pod           #   Nome do recurso
  namespace: laboratorio  #   Em qual namespace ele será criado
spec:                     # Especificação: O QUE você quer que o recurso seja/faça
  # (conteúdo depende do tipo de recurso)
```

Esses quatro campos de topo (`apiVersion`, `kind`, `metadata`, `spec`) estão presentes em **praticamente todo recurso** Kubernetes. É a estrutura que você vai ver repetidamente.

## Seção 8 — Criando um Pod via manifesto YAML

Agora vamos combinar tudo: a sintaxe YAML com o conceito de Pod.

### Laboratório 4 — Pod com manifesto YAML

**1. Criar o arquivo do manifesto:**

Crie um arquivo chamado `pod-nginx.yaml` com o seguinte conteúdo:

```yaml
apiVersion: v1              # Pods usam a API v1 (a mais básica)
kind: Pod                   # Tipo de recurso: Pod
metadata:
  name: nginx-yaml          # Nome do Pod
  namespace: laboratorio    # Namespace de destino
  labels:                   # Rótulos (etiquetas) para organização e seleção
    app: webserver           #   Rótulo chave=valor (podemos inventar o que quiser)
spec:
  containers:               # Lista de contêineres dentro do Pod
    - name: nginx            # Nome do contêiner (para identificação nos logs)
      image: nginx:1.27      # Imagem e versão (tag) do contêiner
      ports:
        - containerPort: 80  # Porta que o contêiner expõe (informativo)
```

> 💡 **Sobre labels (rótulos):** Labels são etiquetas chave-valor que você gruda nos seus recursos. Elas não mudam o comportamento do recurso — servem para **organizar e selecionar**. Mais adiante, veremos como Deployments usam labels para saber quais Pods pertencem a eles.

**2. Aplicar o manifesto:**

```bash
kubectl apply -f pod-nginx.yaml
# "apply -f" lê o arquivo YAML e cria (ou atualiza) o recurso no cluster.
# Esta é a forma padrão de criar recursos no Kubernetes.
```

**3. Verificar:**

```bash
kubectl get pods -n laboratorio
# O Pod "nginx-yaml" deve aparecer com status "Running".
```

**4. Ver os labels do Pod:**

```bash
kubectl get pods -n laboratorio --show-labels
# A coluna LABELS mostrará: app=webserver
```

**5. Filtrar Pods por label:**

```bash
kubectl get pods -n laboratorio -l app=webserver
# O flag "-l" filtra recursos por label.
# Mostra apenas Pods que tenham o label app=webserver.
```

**6. Limpar:**

```bash
kubectl delete -f pod-nginx.yaml
# Remove o recurso descrito no arquivo.
# Assim como "apply -f" cria a partir do arquivo, "delete -f" remove.
```

## Seção 9 — O problema de Pods soltos

Nos laboratórios anteriores, você criou Pods diretamente. Isso funciona, mas tem problemas sérios:

- **Se o Pod falhar, ninguém o recria.** Ele simplesmente desaparece.
- **Se você quiser 3 cópias, precisa criar 3 Pods manualmente.**
- **Se quiser atualizar a versão da imagem, precisa deletar e recriar cada Pod.**

Lembra da analogia do gerente de hotel? Criar Pods manualmente é como o hóspede ir ele mesmo limpar os quartos. Funciona, mas derrota o propósito de ter um gerente.

É aqui que entram os **Deployments**.

## Seção 10 — Deployments: gerenciando Pods de forma inteligente

### O que é um Deployment?

Um Deployment é um recurso do Kubernetes que **gerencia Pods por você**. Quando você cria um Deployment, está dizendo ao Kubernetes:

*"Quero que **N cópias** deste Pod estejam sempre rodando. Se uma morrer, crie outra. Se eu mudar a versão da imagem, atualize gradualmente."*

### A hierarquia: Deployment → ReplicaSet → Pod

Quando você cria um Deployment, o Kubernetes na verdade cria uma cadeia de três recursos:

```
Deployment (o que você cria)
  └── ReplicaSet (criado automaticamente pelo Deployment)
        └── Pod (criado automaticamente pelo ReplicaSet)
        └── Pod
        └── Pod
```

- **Deployment** — declara o estado desejado (qual imagem, quantas réplicas). É o que você cria e gerencia.
- **ReplicaSet** — garante que o número correto de Pods esteja sempre rodando. É criado automaticamente pelo Deployment. Você raramente interage com ele diretamente.
- **Pod** — a unidade que efetivamente roda o contêiner. Criado e gerenciado pelo ReplicaSet.

> 💡 **Na prática:** você interage com o Deployment. O ReplicaSet e os Pods são consequência. Pense no Deployment como seu ponto de controle.

## Seção 11 — Criando um Deployment

### Laboratório 5 — Primeiro Deployment

**1. Criar o arquivo `deployment-nginx.yaml`:**

```yaml
apiVersion: apps/v1                # Deployments usam a API "apps/v1"
kind: Deployment                   # Tipo de recurso: Deployment
metadata:
  name: nginx-deployment           # Nome do Deployment
  namespace: laboratorio
spec:
  replicas: 3                      # Quantidade de Pods desejados
  selector:                        # Como o Deployment encontra seus Pods
    matchLabels:
      app: nginx-lab               #   "Gerencie Pods que tenham este label"
  template:                        # Modelo (template) para criar os Pods
    metadata:
      labels:
        app: nginx-lab             #   Label dos Pods (DEVE coincidir com selector)
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80
```

Vamos destrinchar o que há de novo aqui:

- **`replicas: 3`** — queremos 3 cópias (Pods) rodando simultaneamente
- **`selector.matchLabels`** — é como o Deployment identifica quais Pods pertencem a ele. Funciona como um filtro: "eu gerencio os Pods que tenham o label `app: nginx-lab`"
- **`template`** — é o "molde" usado para criar cada Pod. Note que ele tem sua própria seção `metadata` e `spec`, exatamente como o manifesto de Pod que fizemos antes
- **Os labels no `template.metadata.labels` DEVEM coincidir** com os do `selector.matchLabels`. Se não coincidirem, o Deployment não conseguirá encontrar seus próprios Pods, e o Kubernetes rejeitará o manifesto

**2. Aplicar:**

```bash
kubectl apply -f deployment-nginx.yaml
# Cria o Deployment, que automaticamente cria o ReplicaSet e os 3 Pods.
```

**3. Verificar o Deployment:**

```bash
kubectl get deployments -n laboratorio
# READY 3/3 significa: 3 Pods desejados, 3 Pods prontos.
```

**4. Verificar o ReplicaSet (criado automaticamente):**

```bash
kubectl get replicasets -n laboratorio
# Você verá um ReplicaSet com um nome como "nginx-deployment-xxxxxxxxx".
# O sufixo aleatório é gerado pelo Kubernetes.
# DESIRED=3, CURRENT=3, READY=3 → tudo certo.
```

**5. Verificar os Pods (criados automaticamente pelo ReplicaSet):**

```bash
kubectl get pods -n laboratorio
# Você verá 3 Pods com nomes como:
#   nginx-deployment-xxxxxxxxx-abcde
#   nginx-deployment-xxxxxxxxx-fghij
#   nginx-deployment-xxxxxxxxx-klmno
# O padrão é: nome-do-deployment + hash-do-replicaset + hash-aleatório
```

**6. Ver tudo junto:**

```bash
kubectl get all -n laboratorio
# Mostra Deployments, ReplicaSets e Pods em uma única listagem.
# Útil para ver a hierarquia completa.
```

## Seção 12 — O Deployment em ação: auto-recuperação

A grande vantagem do Deployment é que ele **mantém o estado desejado**. Vamos provar isso.

### Laboratório 6 — Testando a auto-recuperação

**1. Liste os Pods e copie o nome de um deles:**

```bash
kubectl get pods -n laboratorio
# Anote o nome completo de um dos Pods (ex: nginx-deployment-xxxxxxxxx-abcde)
```

**2. Delete esse Pod manualmente:**

```bash
kubectl delete pod <NOME_DO_POD> -n laboratorio
# Substitua <NOME_DO_POD> pelo nome que você copiou.
# Isso simula uma falha — o Pod foi destruído.
```

**3. Imediatamente, liste os Pods novamente:**

```bash
kubectl get pods -n laboratorio
# Observe: ainda há 3 Pods!
# Um deles é novo (nome diferente, idade menor).
# O Kubernetes detectou que havia apenas 2 Pods rodando
# e criou um novo para manter as 3 réplicas desejadas.
```

Isso é o Kubernetes agindo como o gerente de hotel: você pediu 3 quartos, um ficou indisponível, ele imediatamente providenciou outro.

## Seção 13 — Escalando o Deployment

Mudar o número de réplicas é simples.

### Laboratório 7 — Escalando para cima e para baixo

**1. Escalar para 5 réplicas:**

```bash
kubectl scale deployment nginx-deployment \
  --replicas=5 \
  -n laboratorio
# Altera o número desejado de réplicas de 3 para 5.
```

**2. Verificar:**

```bash
kubectl get pods -n laboratorio
# Agora você deve ver 5 Pods rodando.
```

**3. Escalar de volta para 2:**

```bash
kubectl scale deployment nginx-deployment \
  --replicas=2 \
  -n laboratorio
```

**4. Verificar novamente:**

```bash
kubectl get pods -n laboratorio
# 3 Pods foram terminados. Apenas 2 permanecem.
# O Kubernetes removeu os Pods excedentes automaticamente.
```

**5. Também é possível escalar editando o YAML:**

Abra o arquivo `deployment-nginx.yaml`, altere `replicas: 2` para `replicas: 4`, e aplique novamente:

```bash
kubectl apply -f deployment-nginx.yaml
# O Kubernetes compara o estado atual com o arquivo
# e faz apenas as mudanças necessárias (adiciona 2 Pods).
```

## Seção 14 — Atualizando a versão da imagem

Outra tarefa comum é atualizar a versão da aplicação. O Deployment faz isso de forma gradual (**rolling update**): ele cria Pods novos com a nova versão e só destrói os antigos quando os novos estiverem prontos.

### Laboratório 8 — Rolling Update

**1. Atualizar a imagem do Nginx de 1.27 para a versão latest:**

```bash
kubectl set image deployment/nginx-deployment \
  nginx=nginx:latest \
  -n laboratorio
# "nginx=nginx:latest" significa:
#   contêiner chamado "nginx" → nova imagem "nginx:latest"
# O nome do contêiner é o que definimos no campo "name" do YAML.
```

**2. Acompanhar o rollout:**

```bash
kubectl rollout status deployment/nginx-deployment -n laboratorio
# Mostra o progresso da atualização em tempo real.
# Quando disser "successfully rolled out", todos os Pods
# foram atualizados para a nova versão.
```

**3. Verificar que os Pods foram recriados:**

```bash
kubectl get pods -n laboratorio
# Os Pods terão nomes novos (novos hashes), indicando
# que são Pods diferentes dos anteriores.
```

**4. Verificar os ReplicaSets:**

```bash
kubectl get replicasets -n laboratorio
# Você verá DOIS ReplicaSets:
#   - O antigo (com 0 Pods)
#   - O novo (com os Pods atuais)
# O Kubernetes mantém o antigo para possibilitar rollback.
```

**5. Desfazer a atualização (rollback):**

```bash
kubectl rollout undo deployment/nginx-deployment -n laboratorio
# Reverte para a versão anterior.
# O ReplicaSet antigo volta a ter Pods, e o novo fica com 0.
```

**6. Confirmar o rollback:**

```bash
kubectl describe deployment nginx-deployment -n laboratorio | grep Image
# Deve mostrar "nginx:1.27" novamente.
```

## Limpeza final

Quando terminar o laboratório, remova todos os recursos:

```bash
# Deletar o Deployment (que deleta automaticamente o ReplicaSet e os Pods)
kubectl delete -f deployment-nginx.yaml

# Deletar o namespace (e tudo que restou dentro dele)

```

## Resumo do que você aprendeu

| Conceito | O que é | Comando principal |
|---|---|---|
| **Cluster** | O ambiente Kubernetes como um todo | (habilitado no Docker Desktop) |
| **kubectl** | Ferramenta de linha de comando | `kubectl <comando>` |
| **kubeconfig** | Arquivo com credenciais e endereços de clusters | `kubectl config view` |
| **Contexto** | Atalho para cluster + usuário + namespace | `kubectl config current-context` |
| **Namespace** | Divisão lógica dentro do cluster | `kubectl get namespaces` |
| **Pod** | Menor unidade executável (envelope para contêineres) | `kubectl get pods` |
| **Manifesto YAML** | Arquivo declarativo que descreve um recurso | `kubectl apply -f arquivo.yaml` |
| **Deployment** | Gerencia Pods com réplicas e atualizações | `kubectl get deployments` |
| **ReplicaSet** | Mantém o número correto de Pods (criado pelo Deployment) | `kubectl get replicasets` |
| **Scaling** | Alterar o número de réplicas | `kubectl scale deployment ...` |
| **Rolling Update** | Atualização gradual da versão da imagem | `kubectl set image ...` |
| **Rollback** | Desfazer uma atualização | `kubectl rollout undo ...` |
